# GRAIL Workshops Website

This repository hosts the **GRAIL workshop series** website (e.g., “GRAIL‑V @ CVPR 2026”), built with **Jekyll** and deployed via **GitHub Pages**.

Key docs
- `EDITING_GUIDE.md`: organizer-friendly instructions for updating content.
- `AGENTS.md`: repo conventions for contributors and code agents.

## Project Structure

- Pages: `index.md`, `cfp.md`, `program.md`, `speakers.md`, `venue.md`, `faq.md`, `updates.md`
- Content data (edit here first): `_data/*.yml`
  - `_data/site.yml`: hero, branding, loop content
  - `_data/speakers.yml`, `_data/organizers.yml`, `_data/committee.yml`
  - `_data/submission.yml`, `_data/dates.yml`, `_data/program.yml`, `_data/navigation.yml`
- Templates: `_layouts/` (global layout) and `_includes/` (components; sections in `_includes/sections/`)
- Assets: `assets/css/`, `assets/js/`, images in `assets/` and headshots in `assets/people/`
- Search index: `search.json` (generated by Jekyll; used by the site search modal)

## Local Development

Prereqs: Ruby + Bundler.

```bash
bundle install
bundle exec jekyll serve --config _config.yml,_config.dev.yml
```

Then open the printed local URL. The dev config removes `baseurl` so routes look like production.

## Common Content Updates

- Update navigation: `_data/navigation.yml`
- Add/edit a speaker: `_data/speakers.yml` (links + optional `photo`)
  - Put headshots in `assets/people/` and reference as `photo: "/assets/people/<file>"`
- Update deadlines/timeline: `_data/dates.yml` and/or `_data/submission.yml`
- Update the CFP text: `cfp.md` + `_data/submission.yml`

## Deployment (GitHub Pages)

- Deploys on push to `main` via `.github/workflows/jekyll-gh-pages.yml`.
- Current production URL: `https://grailworkshops.github.io/` (org-site). `_config.yml` must keep `baseurl: ""`.
- When linking in Liquid templates, use `{{ "/path/" | relative_url }}` (except pure `#anchors`) to avoid broken links if the base URL ever changes.

## Contributing

- Prefer content changes in `_data/*.yml` and `*.md`; keep templates minimal and reusable.
- Include screenshots for UI changes (desktop + mobile) and verify: nav links, CSS/JS load, and search works.
- Commit messages are short and imperative (optionally scoped), e.g. `UX: ...`, `Site: ...`, `Fix ...`.
